# Seed Data Analysis

## Current Behavior

### Entry Point

`seed_data.py` exposes three commands:

| Command | Behavior |
|---|---|
| `seed` | Inserts all data only if the DB is empty (categories count = 0). Skips silently if already seeded. |
| `reseed` | Calls `clear()` then re-runs the full insert. Aborts if any occurrences are still linked to Google Calendar. |
| `cards` | Inserts credit cards only; skips if any card rows exist. |

### `clear()` — What It Does Today

```python
def clear(db):
    db.query(Occurrence).delete()   # bulk DELETE — wipes all rows including gcal_event_id
    db.query(Event).delete()
    db.query(CreditCard).delete()
    db.query(Category).delete()
    db.commit()
```

This is a **full table wipe** in dependency order. Every row in every table is deleted, including:
- All `gcal_event_id` and `synced_at` values on occurrences
- All user-created events (not just seeded ones)
- All user-created credit cards (not just seeded ones)
- All occurrence status history (completed, skipped notes)

### `seed()` — What It Does Today

After `clear()`, it runs a plain INSERT for each entity:

1. **Categories** — inserts all entries from `CATEGORIES` list (now sourced from `config.yml`)
2. **Events** — inserts all from `EXAMPLE_EVENTS` + fetched MLB/NBA/NHL schedules; no deduplication
3. **Credit Cards** — inserts all from `CREDIT_CARDS` list; auto-creates close/due/fee events via `ensure_card_events()`
4. **Occurrences** — generated by `generate_all_occurrences()` and `generate_credit_card_occurrences()`

## Problems with Current Behavior

### 1. No upsert — only wipe and re-insert

Every `reseed` destroys ALL data and starts fresh. There is no concept of "this record already exists and is unchanged — skip it."

### 2. User-created data is silently destroyed

Any event, credit card, or category the user created manually through the API is deleted by `clear()`. The seed has no way to distinguish seeded records from user records.

### 3. Google Calendar sync linkage is broken on every reseed

`gcal_event_id` lives on `Occurrence` rows. When `clear()` bulk-deletes occurrences, all GCal linkage is lost. New occurrences get new primary key IDs. The next sync inserts brand-new GCal events instead of updating existing ones, creating duplicates in Google Calendar. (A guard was added to `clear()` to block this, but it forces the user to wipe GCal before every reseed.)

### 4. No identity across reseeds

Events and credit cards have no stable natural key enforced by a unique DB constraint. After a reseed, the same event (e.g., "Birthday: Johnny") gets a new primary key. Any external references (GCal `calendarAppId`, occurrence foreign keys) are invalidated.

### 5. Occurrence state is lost

Completed/skipped status, notes, and GCal sync state on occurrences are all deleted. A reseed has no way to preserve the fact that "the user already marked this occurrence as completed."

---

## Desired Behavior

The seed should behave like a reconciliation, not a wipe-and-replace:

| Scenario | Desired Action |
|---|---|
| Record in seed data, not in DB | INSERT |
| Record in seed data, in DB, no field changes | Skip (leave untouched) |
| Record in seed data, in DB, fields changed | UPDATE only changed fields |
| Record in DB, not in seed data, was created by seed | DELETE |
| Record in DB, not in seed data, created by user | Leave alone |

---

## Natural Keys (How to Identify "Same" Record)

Each entity needs a stable natural key — the field(s) that make a record unique across reseeds regardless of primary key.

| Entity | Natural Key | Notes |
|---|---|---|
| `Category` | `name` | Already `UNIQUE NOT NULL` in DB — safe to use |
| `Event` | `title` | Not currently unique in DB; seeded titles are distinct strings — add `UNIQUE` constraint on `(title)` for seeded events, or add an `is_seeded` flag |
| `CreditCard` | `name` + `issuer` | No unique constraint today; combination should be unique in practice |
| `Occurrence` | `event_id` + `occurrence_date` | Already has `UNIQUE` constraint (`uq_event_occurrence_date`) — safe |

## Seeded vs User-Created Tracking

Currently there is **no flag** in the DB to mark a record as seeded. Without one, the seed cannot safely delete records that are absent from seed data, because it cannot tell whether a record was seeded (and then removed from seed data) or user-created.

### Recommended Approach: `is_seeded` boolean column

Add `is_seeded = Column(Boolean, default=False)` to `Category`, `Event`, and `CreditCard`.

- Set to `True` on every insert/update performed by seed_data.py
- Leave `False` for records created via the API (user-created)
- The reconciliation delete step only removes records where `is_seeded = True` AND the title/name is not present in the current seed data

### Alternative: Infer from title match

Without a DB schema change, the seed could define that any `Event` whose `title` exactly matches a seed entry is "owned by seed." Records with titles not in the seed list are treated as user-created and left alone. This is fragile if a user creates an event with the same title as a seed entry.

## Reconciliation Algorithm (Proposed)

```
function reconcile_seed():

  # ── Categories ───────────────────────────────────────────────────────
  existing_cats = {cat.name: cat for cat in db.query(Category)}
  for data in CATEGORIES:
    if data.name not in existing_cats:
      INSERT Category(data)
    elif fields_differ(existing_cats[data.name], data):
      UPDATE existing_cats[data.name] with changed fields only
    else:
      SKIP

  for cat in existing_cats.values():
    if cat.is_seeded and cat.name not in seed_cat_names:
      DELETE cat  # removed from seed data

  # ── Events ───────────────────────────────────────────────────────────
  existing_events = {ev.title: ev for ev in db.query(Event).filter(Event.credit_card_id.is_(None))}
  for (title, cat_name, rrule, dtstart, ...) in SEED_EVENTS:
    if title not in existing_events:
      INSERT Event(...)
      generate_occurrences(db, new_event)
    elif fields_differ(existing_events[title], ...):
      UPDATE existing_events[title] with changed fields
      if recurrence fields changed (rrule, dtstart, dtend_rule):
        regenerate_occurrences(db, existing_events[title])
    else:
      SKIP

  for ev in existing_events.values():
    if ev.is_seeded and ev.title not in seed_event_titles:
      DELETE ev  # cascades to occurrences

  # ── Credit Cards ─────────────────────────────────────────────────────
  existing_cards = {(c.name, c.issuer): c for c in db.query(CreditCard)}
  for card_data in CREDIT_CARDS:
    key = (card_data.name, card_data.issuer)
    if key not in existing_cards:
      INSERT CreditCard(card_data)
      ensure_card_events(db, new_card, cc_cat_id)
      generate_credit_card_occurrences(db, new_card)
    elif fields_differ(existing_cards[key], card_data):
      UPDATE existing_cards[key] with changed fields
      ensure_card_events(db, existing_card, cc_cat_id)  # add any missing events
      generate_credit_card_occurrences(db, existing_card)
    else:
      SKIP

  for card in existing_cards.values():
    if card.is_seeded and (card.name, card.issuer) not in seed_card_keys:
      DELETE card  # cascades to events and occurrences
```

## Impact on Google Calendar Sync

Under the proposed reconciliation approach:

| Scenario | Effect on GCal |
|---|---|
| No field changes | `gcal_event_id` preserved; no GCal API call needed |
| Non-recurrence field changed (description, priority) | `gcal_event_id` preserved; next sync updates the GCal event in-place |
| Recurrence field changed (rrule, dtstart) | Old occurrences deleted (cascade), new ones generated; `gcal_event_id` lost on replaced occurrences — next sync inserts new GCal events |
| Event removed from seed | Row deleted (cascade to occurrences); GCal events become orphaned — a sync cleanup step should delete them |
| New event added to seed | New row inserted; `gcal_event_id` = NULL until next sync |

## Summary

| | Today | Desired |
|---|---|---|
| First run | Full insert | Full insert |
| Re-run, no changes | Skip (all or nothing) | Per-record skip |
| Re-run, some fields changed | Wipe + re-insert everything | Per-record update |
| User-created record | Deleted on reseed | Preserved |
| GCal linkage on reseed | Destroyed | Preserved for unchanged records |
| Removed seed entry | N/A (wipe removes everything) | Deleted only if `is_seeded = True` |
